The first critical phase of this project was to replicate the original RISCV-Matrix execution environment and rebuild the benchmark suite under the current toolchain. The historical speed results were generated using an older cross-compiler toolchain and test suite configuration. This section compares those original results, designated the Grace baseline (commit \texttt{113f4c66}, December 2022), against the current, optimized pipeline (February 2026). All cycle counts were measured on the Spike RISC-V ISA simulator with a vector length of VLEN=1024.

Next, the focus was to benchmark the ts-traversal translation pipeline with various MATLAB and Octave tests for correctness and speed. Finally, some non-vectorized functions from RISCV-Matrix were vectorized and mapped onto the ts-traversal pipeline, ensuring speed-ups from the MATLAB to the RISC-V binary outputs.

\section{Additional Vectorization}

\section{Total Runtime Comparison}

Table~\ref{tab:total-runtime} details the total runtime comparison across thirty distinct operational tests, highlighting the specific cycle counts and the corresponding percentage changes in execution speed.

\input{tables/table_total_runtime}

An analysis of the data presented in Table~\ref{tab:total-runtime} reveals several critical insights regarding the current state of the compilation pipeline. Across the entire test suite, the overall execution time decreased by approximately 3.9 percent, dropping from 419.50 Giga-cycles to 403.01 Giga-cycles. However, the distribution of these performance gains is highly non-uniform. The majority of the standard arithmetic and logical operations, such as ceiling calculations, logical ANDs, and greater-than-or-equal comparisons, experienced moderate speedups ranging from ten to twenty percent. This systemic improvement is directly attributable to significant enhancements within the underlying RISC-V GNU cross-compiler toolchain. The newer iterations of the GCC compiler generate substantially tighter scalar and vector code, featuring optimized register allocation algorithms and superior instruction scheduling that were unavailable during the generation of the historical baseline.

The four core arithmetic tests---\texttt{minus} ($-17.3\%$), \texttt{plus} ($-17.3\%$), \texttt{times} ($-16.4\%$), and \texttt{scale} ($-10.9\%$)---exhibited reductions consistent with the broader toolchain-driven improvements, confirming that the GCC updates benefit both vectorized and scalar code paths uniformly across these operations.

The four comparison tests utilizing Octave-mode semantics---\texttt{octave\_ge}, \texttt{octave\_gt}, \texttt{octave\_le}, and \texttt{octave\_lt}---showed negligible improvement ($< 1\%$). These tests dominate the total runtime at approximately 51 Giga-cycles each (versus $\sim$3.6 Giga-cycles for their MATLAB-mode counterparts) because Octave comparison semantics for complex numbers require computing the magnitude ($|z| = \sqrt{\text{re}^2 + \text{im}^2}$) for both operands and, when magnitudes are equal, resolving ties via the phase angle ($\text{atan2}(\text{im}, \text{re})$). The phase-angle path (\texttt{arg}) currently executes as a scalar libc call because it was never mapped to RVV intrinsics, making it the dominant cost. In contrast, MATLAB-mode comparisons only compare the real parts of complex numbers, requiring a single vector comparison instruction. Their dominance skews the aggregate improvement figure downward; excluding them, the remaining 26 tests show a collective improvement of 7.2 percent.

\subsection{The \texttt{divide} Anomaly}

The \texttt{divide} test is the only operation that exhibited a total runtime \emph{increase} ($+3.0\%$, from 5.47 to 5.64 Gcycles). An investigation of the per-type-combination cycle counts at $N = 1{,}050{,}625$ revealed that this regression is localized to the complex division reference implementations. The reference cycle counts for complex-involving type combinations approximately doubled between Grace and current (e.g., \texttt{int$\times$complex ref}: 37.1M $\rightarrow$ 74.9M cycles; \texttt{double$\times$complex ref}: 36.1M $\rightarrow$ 73.9M cycles), while the vectorized cycle counts remained stable ($\sim$8M cycles). This suggests that the updated GCC toolchain generates less efficient scalar code for complex division specifically---likely due to changes in how the compiler handles the multi-step real/imaginary arithmetic required for complex number division (\texttt{(a+bi)/(c+di)}). This simultaneously \emph{improved} the vec/ref ratio for those combinations (e.g., \texttt{int$\times$complex}: 0.216 $\rightarrow$ 0.106), because the reference denominator grew while the vectorized numerator remained constant.

Conversely, the \texttt{int$\times$int} reference for \texttt{divide} dropped from 14.4M to 4.9M ($-66\%$), indicating that the compiler dramatically improved integer-only scalar division. This demonstrates that the same toolchain update can simultaneously improve and degrade performance across different data type combinations within a single function.

\subsection{Ratio Shifts Due to Compiler Improvements}

While the vectorized implementations maintained approximately constant cycle counts between Grace and current, several vec/ref ratios increased significantly---meaning the scalar \emph{reference} code got faster, narrowing the vectorization advantage. Three distinct patterns were identified:

\paragraph{Pattern 1: Integer memory operations.} For operations that perform simple element-wise copies or integer manipulation---\texttt{ceil}, \texttt{floor}, \texttt{round}, \texttt{full\_duplicate}, and \texttt{ones}---the integer reference cycle counts dropped uniformly from $\sim$3.2M to $\sim$2.2M (a 31\% improvement). The vectorized counts remained at $\sim$2.0M. This caused the integer ratio to jump from $\sim$0.625 to $\sim$0.910, nearly reaching parity. The compiler likely now generates optimized memory copy or store sequences (e.g., using wider loads/stores) that approach vector performance for these operations.

\paragraph{Pattern 2: Logical operations on non-integer types.} For \texttt{not} and \texttt{true}, the double and complex reference counts dropped by $\sim$65\% (e.g., \texttt{not} double ref: 10.3M $\rightarrow$ 3.6M). The vectorized counts stayed at $\sim$3.1M. This caused the double/complex ratios to jump from $\sim$0.30 to $\sim$0.85. These functions perform type-conversion-heavy logic that the updated GCC compiler now optimizes far more aggressively.

\paragraph{Pattern 3: Binary operations.} Several binary operations exhibited similar compiler-driven ref improvements for specific type combinations. The \texttt{and} (dbl$\times$dbl), \texttt{matlab\_ge} through \texttt{matlab\_lt} (dbl$\times$dbl), and \texttt{octave\_ge} through \texttt{octave\_lt} (dbl$\times$dbl) all showed double reference speedups of 3--5$\times$, pushing their ratios from $\sim$0.25 to $\sim$0.83. The full details are available in the Appendix (Table~\ref{tab:ratio-binary-full}).

In all cases, the vectorized code did \emph{not} get slower---the reference code simply got faster. This means the absolute performance of the library improved or held steady, but the \emph{relative} advantage of explicit vectorization narrowed for these specific combinations.

\subsection{The \texttt{isEqual} Anomaly}

The \texttt{isEqual} test consistently returns a vec/ref ratio greater than 1.0 for complex type combinations (e.g., complex$\times$complex: 1.172 in Grace, 1.121 in current), indicating that the vectorized implementation is slower than scalar. Investigation revealed that this is a \emph{test harness artifact} caused by short-circuit evaluation.

The evidence shows that the raw cycle counts for \texttt{isEqual} complex$\times$complex remain \textbf{exactly 75 cycles (vec) and 64 cycles (ref) at every single matrix size tested}---from $N=1$ through $N=1{,}050{,}625$:

If the implementations were actually traversing the data, the cycle counts would scale linearly with $N$. The fact that they remain locked at 75/64 cycles regardless of input size proves that both implementations exit after comparing only the \emph{first element}. An examination of the test data generation confirmed that the \texttt{isEqual} test compares matrices populated with entirely divergent data streams, guaranteeing an immediate mismatch.

The 17\% performance gap (75 vs.\ 64 cycles) is attributable to \emph{vector setup overhead}: the vectorized path must execute \texttt{vsetvl} (vector length configuration) and stride-loading instructions before performing comparisons. The scalar path directly loads and compares two values. In realistic workloads where equality must be verified across a full dataset traversal, the vectorized implementation vastly outperforms scalar execution, as evidenced by the complex$\times$double combination achieving a ratio of 0.015 (66$\times$ speedup).

In all three cases, the vectorized cycle counts remained approximately constant ($\sim$5.3M for \texttt{and}, $\sim$5.4M for \texttt{or}). The regressions are entirely driven by GCC's updated optimizer reducing the reference cycle counts by a factor of five to six. Disassembling the reference-kernel binaries with \texttt{riscv64-unknown-elf-objdump} reveals the mechanism: the inner loop of the reference implementation for both \texttt{and} and \texttt{or} on double-valued inputs now compiles to a vector-length-agnostic sequence of \texttt{vsetvli\,a2,s0,e64,m1,ta,ma} followed by a masked \texttt{vle64.v}, \texttt{vmfne.vv} compare-to-zero, \texttt{vmand.mm}/\texttt{vmor.mm}, and \texttt{vse32.v} write-back---exactly the pattern one would hand-write. In other words, the ``scalar'' reference is no longer scalar: GCC 15's RISC-V backend auto-vectorizes it at \texttt{-O2} \cite{gcc14-release,gcc15-release}, and the resulting code happens to beat the library's hand-written \texttt{m8} kernel because it fuses the compare and mask steps that the intrinsic version keeps separate.

This has two implications. First, compiler evolution can erode the benefits of manual vectorization if the hand-written code is not periodically re-optimized. Second, the hand-written kernels here use \texttt{LMUL=8}, which the RISCV-Matrix README warns will ``cause extra loads and stores to be called implicitly'' when the kernel uses more than a handful of live vector registers. The README's own sizing rule---``LMUL should be 32 divided by the smallest power of two greater than or equal to the number of vector variables used in the function''---prescribes \texttt{m4} or \texttt{m2} for kernels that load two operands, produce a mask, combine masks, and write a result, since that register pressure exceeds what \texttt{m8} can hold without spilling. The autovectorized reference sticks to \texttt{m1} and sidesteps the spill entirely; the new \texttt{meanM\_vec}/\texttt{varM\_vec} fast paths added this semester also use \texttt{m1} by design. Revisiting the LMUL choice \emph{downward} for the legacy \texttt{andM}/\texttt{orM}/\texttt{notM} kernels---rather than raising it---is therefore the recovery path most consistent with both the README's written guidance and the observed disassembly evidence: the measured regression is a spill problem, not a throughput problem.

\section{TS-traversal Compiler}

Beyond the resolution of benchmarking anomalies, significant debugging efforts have focused on addressing algorithmic failures within the ts-traversal compiler phase. When the project was inherited, the TypeScript traversal would emit C that compiled but failed on several fundamental constructs: higher-dimensional indexing (3-D and 4-D array subscripts), various standalone issues that propagated across multiple functions like off-by-one for loop iteration, missing column vector support, missing function mapping, and row major vs. column major representation across MATLAB and C. There were also some tests that were producing different results across the MATLAB and generated C files due to a difference in the type of BLAS (Basic Linear Algebra Subprogram) libraries used between MATLAB (OPENBLAS) and our bare metal RISC-V binaries (LAPACK, RISCVBLAS). The tolerance level has now been set to $5*10^{-10}$ for both absolute and mantissa level checking. After various fixes, each of these paths now round-trips a representative MATLAB script through translation, compilation with \texttt{riscv64-unknown-elf-gcc} 15.2.0, and execution on Spike without harness-level failure.
